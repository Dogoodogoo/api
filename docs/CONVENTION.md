# 🐕 '어디가개' 프로젝트 가이드라인

본 문서는 프로젝트의 일관성 있는 협업과 보안 준수를 위한 브랜치 및 커밋 규칙을 정의합니다.

---

## 🌳 1. 브랜치 관리 전략 (Git Flow)

프로젝트의 안정적인 배포와 개발 효율을 위해 브랜치를 논리적으로 분리합니다.

| 브랜치 종류 | 설명 | 관리 규칙 |
| :--- | :--- | :--- |
| **main** | 제품 출시용 코드 | 관리자(PL)만 병합 가능하며 직접 Push를 금지합니다. |
| **develop** | 다음 버전 통합 브랜치 | 모든 기능 개발 브랜치의 종착지입니다. |
| **feature/** | 신규 기능 개발 | `develop`에서 분기하며, 구현 완료 후 PR을 생성합니다. |
| **hotfix/** | 긴급 버그 수정 | `main`에서 직접 분기하여 수정 후 양쪽 모두 병합합니다. |

### 🔍 CS 지식: Git 히스토리의 자료구조 (DAG)
Git의 커밋 히스토리는 **방향성 비순환 그래프(Directed Acyclic Graph, DAG)** 구조를 가집니다. 각 브랜치는 특정 노드를 가리키는 **가변 포인터(Mutable Pointer)**로 작동하며, 브랜치 생성 및 전환은 $O(1)$의 시간 복잡도를 가집니다.

## ✍️ 2. 커밋 메시지 규칙 (Conventional Commits)

변경 이력을 명확히 추적하기 위해 아래 형식을 엄수합니다.

### 📋 커밋 타입 (Type)
- **feat**: 새로운 기능 추가
- **fix**: 버그 수정
- **docs**: 문서 수정 (README, 주석 등)
- **style**: 코드 포맷팅 (로직 변경 없음)
- **refactor**: 코드 리팩토링
- **test**: 테스트 코드 추가/수정
- **chore**: 빌드 업무, 설정 변경 (build.gradle.kts 등)

### 💡 CS 지식: 원자적 커밋 (Atomic Commits)
커밋은 논리적으로 **독립된 최소 단위**로 수행되어야 합니다(원자성 보장). 이는 추후 특정 기능만 되돌리거나(Revert), 이진 탐색 알고리즘을 활용한 버그 추적(Git Bisect) 시 효율성을 극대화합니다.
